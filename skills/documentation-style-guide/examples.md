# Documentation Examples

## Purpose

Provide examples of preferred and non-preferred writing styles.

---

# Example 1 - Feature Description

## Good

Users can now export dashboard data to Microsoft Excel, making it easier to share and analyze information.

### Why it works

- Customer-focused
- Clear benefit
- Concise

---

## Poor

Implemented Excel export functionality using Apache POI.

### Issues

- Technical implementation detail
- No customer value
- Developer-focused language

---

# Example 2 - Performance Improvement

## Good

Dashboard loading times have been reduced for large datasets.

### Why it works

- Specific
- Customer-focused

---

## Poor

Improved performance.

### Issues

- Ambiguous
- No measurable outcome
- No context

---

# Example 3 - Security Enhancement

## Good

Users can now enable multi-factor authentication to provide an additional layer of account security.

### Why it works

- Explains the feature
- Explains the benefit

---

## Poor

Implemented MFA support.

### Issues

- Technical wording
- No customer impact

---

# Example 4 - Resolved Issue

## Good

Resolved an issue that prevented scheduled reports from running successfully.

### Why it works

- Clear issue description
- Customer-facing language

---

## Poor

Fixed scheduler bug.

### Issues

- Internal terminology
- Lacks context

---

# Example 5 - Missing Information

## Source Information

Added support for configuration profiles.

## Preferred Output

### Missing Information

- Customer impact not provided.
- Intended user audience not identified.
- Benefit not specified.

### Why it works

- Does not invent information.
- Flags required clarification.

---

# General Guidance

When choosing between technical accuracy and readability:

1. Preserve factual accuracy.
2. Use customer-focused language.
3. Remove unnecessary implementation details.
4. Follow Microsoft Style Guide principles.
