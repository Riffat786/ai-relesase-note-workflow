# Release Note Writing Rules

## Purpose

This document defines standards for creating customer-facing release notes.

---

# Core Principles

Release notes should explain:

1. What changed
2. Why it matters
3. Who benefits
4. Any known limitations

---

# Content Categories

## New Features

New capabilities available to users.

Example:

Users can now enable multi-factor authentication for additional account security.

---

## Enhancements

Improvements to existing functionality.

Example:

Dashboard loading times have been reduced for large reports.

---

## Resolved Issues

Customer-facing issues that have been corrected.

Example:

Resolved an issue that prevented report exports from completing successfully.

---

## Security Updates

Security-related improvements.

Example:

Users can now configure multi-factor authentication using authenticator applications.

---

## Known Limitations

Document known restrictions or issues.

Example:

Multi-factor authentication currently supports authenticator applications only.

---

# Writing Rules

## Focus on Customer Value

Preferred:

Users can export reports directly to Excel for easier sharing and analysis.

Avoid:

Implemented export functionality using a new reporting framework.

---

## Avoid Technical Implementation Details

Do not expose:

- Internal architecture
- Database changes
- Framework upgrades
- Refactoring activities

Unless specifically requested.

---

## Avoid Unsupported Claims

Avoid:

- Faster
- Better
- Improved
- Enhanced

Unless measurable evidence is provided.

Preferred:

Dashboard loading times have been reduced for reports containing more than 10,000 records.

---

# Information Gaps

When information is incomplete:

Do not guess.

Instead identify:

## Missing Information

- Customer impact not provided.
- User benefit not specified.
- Feature description incomplete.
