# Documentation Style Guide Skill

## Purpose

This skill provides standardized writing guidance for documentation-related content, including release notes, user-facing communications, knowledge base articles, summaries, announcements, and other documentation deliverables.

Its purpose is to ensure consistency, clarity, professionalism, accessibility, and customer focus across all generated content.

This skill serves as the foundation for all documentation-generation and documentation-review skills.

The skill incorporates Microsoft Style Guide (MSTP) principles and documentation best practices.

---

# Core Principles

## Accuracy First

Use only information provided in the source material.

Do not:

* Invent features
* Invent functionality
* Invent customer benefits
* Invent release scope
* Invent technical details
* Invent assumptions

If information is missing, explicitly identify the gap.

---

## Customer-Focused Writing

Describe changes from the user's perspective.

Focus on:

* What changed
* Why it matters
* Customer impact
* User outcomes

Preferred:

Users can now export dashboard results to Microsoft Excel format, making it easier to share and analyze data.

Avoid:

Implemented Excel export functionality using Apache POI libraries.

---

## Clarity

Use clear and concise language.

Preferred:

Search results load faster.

Avoid:

The application has been enhanced to provide improved performance when retrieving search results.

---

## Consistency

Use approved terminology consistently throughout the document.

Avoid using multiple names for the same feature, capability, or product.

---

# Microsoft Style Guide (MSTP) Standards

Apply Microsoft Writing Style Guide principles unless organizational standards explicitly override them.

---

## Write for the User

Focus on user goals and outcomes.

Preferred:

Users can now schedule reports.

Avoid:

A scheduling capability has been implemented.

---

## Use Conversational Language

Write naturally and directly.

Preferred:

You can now configure notification preferences.

Avoid:

Notification preferences can now be configured by the user.

---

## Use Active Voice

Preferred:

Users can filter search results by date.

Avoid:

Search results can be filtered by date.

---

## Use Present Tense

Preferred:

The dashboard displays performance metrics.

Avoid:

The dashboard will display performance metrics.

---

## Be Clear and Concise

Remove unnecessary words and filler content.

Preferred:

Search results load faster.

Avoid:

Search results have been optimized to provide enhanced loading performance.

---

## Use Inclusive Language

Use respectful and inclusive terminology.

Preferred:

* Primary
* Secondary
* Main
* Subordinate
* Allowlist
* Blocklist

Avoid outdated or non-inclusive terminology.

---

# Voice and Tone

## Voice

Professional, informative, helpful, and objective.

## Tone

* Clear
* Friendly
* Neutral
* Customer-focused

Avoid:

* Marketing language
* Sales language
* Exaggerated claims
* Promotional wording

Examples of wording to avoid:

* Revolutionary
* Best-in-class
* Amazing
* Groundbreaking
* Industry-leading

Unless explicitly provided in approved source material.

---

# Writing Standards

## Active Voice

Preferred:

The application now supports multi-factor authentication.

Avoid:

Multi-factor authentication is now supported by the application.

---

## Present Tense

Preferred:

Users can configure notification preferences.

Avoid:

Users will be able to configure notification preferences.

---

## Conciseness

Keep sentences short and direct.

Preferred:

Search results load faster.

Avoid:

The application has been updated in a way that allows search results to load more efficiently.

---

# Release Note Writing Standards

Every release note entry should answer the following questions whenever information is available:

1. What changed?
2. Who is affected?
3. Why does it matter?
4. What benefit does the user receive?

---

## Customer Value Focus

Focus on outcomes rather than implementation.

Preferred:

Users can now export reports directly to Excel for easier sharing and analysis.

Avoid:

Implemented an export service using a new reporting framework.

---

## Technical Detail Management

Avoid unnecessary implementation details in customer-facing documentation.

Include technical details only when:

* Required by the audience
* Explicitly requested
* Relevant to understanding the change

---

# Terminology Standards

## Product Names

Always use official product names.

Do not:

* Invent abbreviations
* Create alternate names
* Modify approved naming conventions

---

## Feature Names

Use feature names exactly as defined by Product Management.

Example:

Use:

Multi-Factor Authentication

Avoid:

MFA Login Feature

unless officially approved.

---

## User References

Preferred:

* Users
* Administrators
* Customers

Avoid:

* End users
* Consumers

unless organizational standards require them.

---

# Formatting Standards

## Headings

Use sentence case.

Preferred:

### New features

### Enhancements

### Resolved issues

Avoid:

### NEW FEATURES

### New Features

---

## Lists

Use numbered lists for procedures.

Example:

1. Open Settings.
2. Select Notifications.
3. Save your changes.

Use bulleted lists for non-sequential information.

Example:

Benefits include:

* Faster reporting
* Improved usability
* Reduced manual effort

---

## User Interface References

When referring to UI elements:

Use bold formatting.

Example:

Select **Export**.

Select **Settings** > **Notifications**.

---

## Dates

Use a consistent date format throughout the document.

Examples:

* 24 June 2026
* 2026-06-24

Do not mix date formats within the same document.

---

# Accessibility Standards

Documentation should be easy to scan and understand.

Use:

* Meaningful headings
* Short paragraphs
* Clear labels
* Descriptive text
* Simple language

Avoid:

* Large blocks of text
* Excessive abbreviations
* Unnecessary jargon

---

# Ambiguity Detection

Flag vague statements.

Examples:

Ambiguous:

Improved performance.

Preferred:

Reduced dashboard loading times for large datasets.

---

Ambiguous:

Enhanced security.

Preferred:

Users can now enable multi-factor authentication for additional account protection.

---

# Information Gap Handling

If required information is missing:

Do not guess.

Instead provide:

## Missing Information

* Customer impact not provided.
* Feature description incomplete.
* User benefit not specified.
* Audience not identified.

---

# Review Checklist

Before content is considered complete, verify:

* Information is accurate.
* Active voice is used.
* Present tense is used.
* Language is customer-focused.
* Terminology is consistent.
* Inclusive language is used.
* Grammar and spelling are correct.
* No unsupported claims are present.
* Ambiguous statements are clarified.
* Missing information is identified.
* MSTP principles are followed.

---

# Success Criteria

Content generated using this skill should:

* Be accurate.
* Be easy to understand.
* Follow MSTP writing principles.
* Maintain consistent terminology.
* Support customer-facing communication.
* Minimize ambiguity.
* Require minimal editing before review.

---

# Skill Rule

If there is a conflict between source information and this style guide:

Always preserve factual accuracy.

Style guidance must never alter the meaning of the source information.
