# Customer-Friendly Language

## Purpose

Transform technical implementation details into customer-focused language.

Provide consistent writing standards for generating customer-facing release notes.

Use this skill whenever creating, editing, or reviewing release notes intended for customers, administrators, or business users.

---

## Guidelines

- Focus on customer value.
- Explain benefits rather than implementation.
- Avoid technical jargon.
- Use clear and concise language.
- Maintain a professional tone.
- Use active voice.
- Keep wording consistent.

## Avoid

- Internal project names
- Database terminology
- API names
- Infrastructure details
- Developer abbreviations
---

## Audience

* Customers
* Administrators
* Business users
* Product stakeholders

Do **not** assume technical knowledge.

---

## Writing Principles

* Write clearly and concisely.
* Focus on customer value rather than technical implementation.
* Use active voice.
* Use consistent terminology.
* Write in complete sentences.
* Keep descriptions brief (1–3 sentences).

---

## Preferred Language

Use phrases such as:

* Users can now...
* Administrators can now...
* You can now...
* Improved...
* Enhanced...
* Resolved an issue where...
* Fixed an issue that...

Focus on outcomes and benefits.

---

## Avoid

Do not mention:

* Azure DevOps
* ServiceNow
* APIs
* Database changes
* Infrastructure work
* Refactoring
* Logging framework
* Deployment details
* Internal project names

Avoid developer-centric language unless absolutely required.

---

## Good Examples

**Instead of**

Implemented MFA backend service.

**Write**

Users can now secure their accounts using multi-factor authentication.

---

**Instead of**

Optimized SQL indexes.

**Write**

Improved application performance for faster search results.

---

## Success Criteria

The release notes:

* Are understandable by non-technical readers.
* Describe customer value.
* Avoid implementation details.
* Use consistent language and tone.
